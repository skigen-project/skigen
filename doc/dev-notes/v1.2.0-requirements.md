# Skigen v1.2.0 — Requirements Specification

**Status:** Draft
**Started:** 2026-06-14
**Predecessor:** [v1.1.0-requirements.md](v1.1.0-requirements.md)
**Standard:** EES (Energy-Efficient Software)

## 1. Release Theme

**EES focus:** Complete the algorithmic gaps left by v1.1.0 and harden Skigen's parity contract.

v1.2.0 is the first post-broadening release. v1.1.0 delivered the major sklearn surface expansion; v1.2.0 should make that surface deeper, more exact, and more production-shaped. The primary work is therefore not a new wave of estimators for its own sake. The release should first close the explicitly deferred v1.1.0 items, then add the highest-value missing sklearn families where they fit Skigen's Eigen-only, header-only design.

### 1.1 Current Implementation Status

As of the first v1.2.0 implementation sprint:

- `doc/api_registry.json` is enforced by `doc/audit_api_registry.py` and a `skigen_audit_registry` CMake target.
- Registry coverage is reconciled for existing public classes: 75 public classes, 74 guide-backed documented entries, 75 example-backed entries, and 8 rendered plot entries.
- Website dependency alerts are resolved locally: `http-proxy-middleware` is overridden to `3.0.7`, `js-yaml` is overridden to `4.2.0`, and `gray-matter` is patched postinstall for the js-yaml 4 API.
- The rendered example gallery now includes PCA, KMeans, SVC decision regions, t-SNE, UMAP, LocalOutlierFactor, CalibratedClassifierCV reliability, and GradientBoostingRegressor training loss.
- `Skigen/Impute` is implemented for dense numeric `SimpleImputer` and `MissingIndicator`.
- `Skigen/KernelRidge` is implemented for dense single-target `KernelRidge` with linear, RBF, polynomial, and sigmoid kernels.
- `Skigen/GaussianProcess` includes dense fixed-hyperparameter `GaussianProcessRegressor` and binary `GaussianProcessClassifier`; kernel composition remains deferred.
- `Skigen/DiscriminantAnalysis` is implemented for dense `LinearDiscriminantAnalysis` and `QuadraticDiscriminantAnalysis` classifiers.
- `Skigen/Anomaly` is implemented for dense `EllipticEnvelope` and `IsolationForest` outlier detection.
- `Skigen::RidgeClassifier` is implemented for dense binary and multiclass classification.
- `Skigen::Perceptron` is implemented as a dense perceptron-loss SGD classifier wrapper.
- `Skigen::NearestNeighbors` is implemented for dense brute-force k-neighbor and radius-neighbor queries.
- `Skigen::RadiusNeighborsClassifier` and `Skigen::RadiusNeighborsRegressor` are implemented for dense brute-force radius-neighbor prediction.
- `Skigen/CrossDecomposition` is implemented for dense NIPALS `PLSRegression` and `CCA`.
- `feature_selection::mutual_info_classif` and `feature_selection::mutual_info_regression` are implemented for dense numeric data and integrated with `SelectKBest`.
- `CalibratedClassifierCV` supports binary and multiclass calibration, including the `ensemble=false` refit path for sigmoid and isotonic calibration.
- `MLPClassifier` and `MLPRegressor` explicitly feature-gate `LBFGS` with tested errors while retaining Adam / SGD support.
- `MDS` supports multiple `n_init` SMACOF restarts with best-stress selection.
- `DecisionTreeRegressor::fit_multi` uses a shared tree with joint multi-target split scoring.
- `RandomForestRegressor` supports multi-output targets via `fit_multi` / `predict_multi` over shared joint trees.
- `GradientBoostingRegressor` implements `absolute_error`, `huber`, and `quantile` losses with line-search step sizing, plus stochastic boosting via `subsample < 1.0`.
- `GradientBoostingClassifier` implements the `exponential` (AdaBoost-style) loss in addition to `log_loss`.
- `LocallyLinearEmbedding` supports the `modified`, `hessian`, and `ltsa` method variants in addition to `standard`.
- `svm::OneClassSVM` is implemented with a dedicated one-class nu-SVM SMO solver, exposing `decision_function`, `score_samples`, `predict`, and `offset_`.
- `svm::NuSVC` is implemented with a two-constraint nu-classification SMO (per-class mass init, feasibility check on `nu`), exposing `decision_function`, `predict`, `support`, and `intercept`.
- `svm::NuSVR` is implemented with a two-phase nu-regression fit that learns `epsilon` from the `1 - nu` residual quantile, exposing `predict`, `score`, `support`, and `epsilon_fitted`.
- `decomposition::PCA` gains a `svd_solver` parameter (`"full"` / `"randomized"`) and a native sparse fit via implicit centering; the Halko-Martinsson-Tropp randomized SVD is factored into a shared `internal::randomized_svd` helper reused by `TruncatedSVD`.
- `ensemble::HistGradientBoosting{Regressor,Classifier}` use a native leaf-wise gradient/hessian histogram tree grower (`internal::HistTree`) with second-order split gains, L2 regularisation, monotonic constraints, and holdout-based early stopping; the classifier additionally supports multiclass softmax boosting.
- `linear_model::QuantileRegressor` is implemented with a native primal-dual interior-point LP solver (`internal::solve_standard_lp`).
- `svm::Detail::SMO` gains a libsvm-style second-order working-set-selection solver (`internal::smo_binary_wss3`); `smo_binary` delegates to it while `smo_binary_platt` preserves the old simplified solver.
- `manifold::TSNE` adds `method="barnes_hut"` (quadtree-accelerated 2-D repulsion via `internal::QuadTree`), `angle`, and `early_exaggeration` parameters.
- `Skigen/Optimization` is implemented for dense unconstrained `NelderMead` minimization.

One planning priority is to clarify and add simplex-based optimization without conflating two different meanings of "simplex". The mne-cpp candidate replacement is a Nelder-Mead direct-search minimizer used for derivative-free nonlinear objectives such as sphere fitting. The scikit-learn hit for "simplex" is instead `QuantileRegressor`, which delegates to SciPy's linear-programming solvers and exposes solver names such as `revised simplex` and `highs-ds`. v1.2.0 therefore tracks these as separate possible additions: a Skigen-native high-performance Nelder-Mead optimizer, and a distinct sklearn-parity quantile-regression / LP track.

### 1.2 Versioning Policy

Skigen still has a small adoption surface, so v1.2.0 keeps the architectural cleanup window open. Breaking changes are allowed when they make the library more beautiful, coherent, sklearn-faithful, high-performance, or easier to maintain long-term.

Allowed breaking-change categories:

- Constructor parameters may be reordered, renamed, added, or removed when the result improves sklearn parity or eliminates awkward pre-1.0-style API baggage.
- Fitted attributes and accessors may change shape or naming when this brings Skigen closer to sklearn, improves multi-output semantics, or removes accidental inconsistencies.
- Module boundaries, internal header layout, estimator base classes, parameter reflection, and type aliases may be refactored if the change clarifies the architecture or removes duplicated machinery.
- Existing v1.1.0 examples and parity fixtures may be updated rather than preserved verbatim when the underlying API is intentionally improved.
- Compatibility shims are optional, not required. Prefer a clean API over permanent aliases while the project is still young.

Every intentional break must still be documented in `CHANGELOG.md` and, when user-facing, in the v1.2.0 migration notes. The release goal is not churn; it is to spend the remaining low-adoption window on the changes that make Skigen feel crisp, principled, and durable.

## 2. Inherited Guardrails

All non-negotiable rules from v1.0.0 and v1.1.0 remain in force:

- Header-only C++23 library; no compiled sources in `Skigen/`.
- Eigen remains the sole required numerical dependency.
- `Eigen::Ref<const MatrixType>` for dense read-only inputs and `Eigen::SparseMatrix` overloads where sklearn admits sparse data.
- CRTP estimator hierarchy; no virtual dispatch on hot estimator paths.
- Strict sklearn parity for names, defaults, fitted attributes, error conditions, and deterministic random-state semantics.
- Zero warnings under `-Wall -Wextra -Wpedantic` on GCC >= 13, Clang >= 17, and MSVC >= 19.38.
- Every new estimator ships with unit tests, parity coverage, one runnable example, and a guide page.

### 2.1 v1.2.0 Quality Gates

| Gate | Requirement |
|---|---|
| **Deferred-first** | Every v1.1.0 known gap listed in §4 must be either implemented or explicitly rejected with a documented reason before new estimator families may block the release. |
| **Parity depth** | Default-parameter parity is no longer sufficient for mature estimators. Each estimator touched in v1.2.0 must add at least one non-default parity fixture per meaningful solver / loss / kernel family. |
| **Sparse honesty** | Sparse overloads must either stay sparse through the fit path or document and test any densification fallback. Silent densification is forbidden for v1.2.0 additions. |
| **Benchmark sweep** | Existing v1.0.0 and v1.1.0 benchmarks must be run on the release runner; any >10% regression requires a fix or a documented exception. |
| **Documentation parity** | Guide pages must state the sklearn reference class, supported parameter subset, parity fixtures, and known parity gaps. |
| **Maintenance review** | After each major implementation batch and before the release candidate, run an architecture / maintainability / performance review. Findings must be fixed or explicitly tracked before release. |

## 4. Deferred v1.1.0 Work Items

### 4.1 svm::NuSVC, svm::NuSVR, svm::OneClassSVM

**Current state:** Public placeholder shells exist and throw from `fit`.

**Requirement:** Implement the nu-SVM and one-class SMO variants so the three estimators become usable sklearn-parity estimators.

| Estimator | sklearn reference | Required solver |
|---|---|---|
| `NuSVC` | `sklearn.svm.NuSVC` | nu-classification SMO with feasibility checks for `nu`. |
| `NuSVR` | `sklearn.svm.NuSVR` | nu-regression SMO with paired alpha variables. |
| `OneClassSVM` | `sklearn.svm.OneClassSVM` | one-class support estimation with offset / `rho_`. |

**Constructor parity:** Preserve sklearn defaults for `nu`, `kernel`, `degree`, `gamma`, `coef0`, `shrinking`, `probability`, `tol`, `cache_size`, `max_iter`, and `random_state` where applicable.

**Fitted attributes:** Expose `support_`, `support_vectors_`, `dual_coef_`, `intercept_`, `n_support_`, `shape_fit_`, and `offset_` for `OneClassSVM`.

**Implementation constraints:**

- Reuse the v1.1.0 kernel policies (`Linear`, `RBF`, `Polynomial`, `Sigmoid`) and cache infrastructure.
- Add sparse overloads for all three estimators; sparse input must not densify kernel rows unnecessarily.
- `predict`, `decision_function`, and `score_samples` for `OneClassSVM` must match sklearn semantics.
- `probability=true` for `NuSVC` uses the existing Platt-scaling helper.

**Tests:** Binary classification, multiclass OvO classification, regression, one-class novelty detection, invalid `nu` bounds, sparse/dense parity, and sklearn parity fixtures for at least `linear` and `rbf` kernels.

### 4.2 svm::Detail::SMO Parity Upgrade

**Current state:** `SVC` and `SVR` use a simplified Platt-1998 SMO solver; libsvm's second-order working-set selection is deferred.

**Requirement:** Upgrade the kernel SVM solver core to match libsvm's working-set selection and shrinking behavior closely enough for tighter sklearn parity.

**Implementation constraints:**

- Factor working-set selection into a reusable internal helper shared by `SVC`, `SVR`, `NuSVC`, `NuSVR`, and `OneClassSVM`.
- Preserve deterministic iteration order for fixed `random_state`.
- Keep kernel-cache memory bounded by `cache_size` MB.
- Existing public SVM APIs remain unchanged.

**Tests:** Compare support-vector counts, intercepts, predictions, and decision functions against sklearn on small deterministic fixtures. Include timeout / `max_iter` behavior.

### 4.3 Native Sparse Tree Split-Finding

**Current state:** `DecisionTreeClassifier` and `DecisionTreeRegressor` accept sparse input but densify before fitting.

**Requirement:** Implement native sparse split-finding for the decision-tree family and reuse it in ensembles.

**Affected estimators:**

- `DecisionTreeClassifier`, `DecisionTreeRegressor`
- `RandomForestClassifier`, `RandomForestRegressor`
- `GradientBoostingClassifier`, `GradientBoostingRegressor`
- `HistGradientBoostingClassifier`, `HistGradientBoostingRegressor` where sparse input is supported

**Implementation constraints:**

- Prefer CSC traversal for feature-wise split scans and CSR traversal for prediction / row access.
- Missing implicit zeros must participate in split evaluation exactly as sklearn treats sparse zeros.
- Avoid materialising dense `n_samples x n_features` matrices.
- Preserve dense-tree behavior bit-for-bit on dense fixtures.

**Tests:** Sparse/dense parity for classification and regression trees, sparse random forest fit/predict parity, memory regression guard on a wide sparse fixture, and error-condition parity for empty sparse input.

### 4.4 neural_network::MLP solver="lbfgs"

**Current state:** `MLPClassifier` and `MLPRegressor` implement `Adam` and `SGD`; `LBFGS` throws.

**Requirement:** Implement deterministic L-BFGS training behind `SKIGEN_ENABLE_LBFGS`, matching sklearn's `solver="lbfgs"` mode.

**Implementation constraints:**

- Use Eigen's unsupported L-BFGS implementation when available, wrapped behind a small Skigen adapter.
- If the feature flag is disabled, the constructor may still accept `LBFGS`, but `fit` must throw a clear feature-gate error.
- Flatten and unflatten weight matrices without copying where feasible.
- Reuse the existing forward / backward pass; do not introduce a second gradient implementation.

**Tests:** Deterministic parity fixtures for `MLPRegressor` and `MLPClassifier`, feature-flag disabled behavior, gradient-shape checks, and convergence on small XOR / regression fixtures.

### 4.5 feature_selection::mutual_info_classif and mutual_info_regression

**Current state:** `SelectKBest` supports `f_classif`, `f_regression`, and `chi2`; `mutual_info_*` is deferred.

**Requirement:** Add sklearn-compatible mutual information score functions and wire them into `SelectKBest`.

**Implementation constraints:**

- Implement k-nearest-neighbor entropy estimators compatible with sklearn's `mutual_info_classif` and `mutual_info_regression` defaults.
- Support continuous and discrete feature masks.
- Preserve sparse support for discrete sparse input where sklearn supports it.
- Clamp negative MI estimates to zero, matching sklearn behavior.

**Tests:** Classification and regression fixtures, discrete-feature mask behavior, sparse text-style fixture, random-state determinism, and `SelectKBest` integration.

### 4.6 manifold::Spectra Integration

**Current state:** Manifold learners use dense Eigen eigensolvers; optional `Spectra` integration is deferred.

**Requirement:** Add optional truncated eigensolver support behind `SKIGEN_ENABLE_SPECTRA`.

**Affected estimators:** `Isomap`, `LocallyLinearEmbedding`, `SpectralEmbedding`, and any future graph-embedding estimator.

**Implementation constraints:**

- `Spectra` remains optional and header-only; Eigen remains the sole required dependency.
- Dense fallback must remain the default when the feature flag is off.
- Solver choice follows sklearn-compatible `eigen_solver="auto" | "arpack" | "dense"` spelling through Skigen enums.
- Parity tests use sign/rotation-invariant checks where eigenvectors are non-unique.

**Tests:** Dense fallback parity, feature-flag compile path, large sparse graph smoke test, and invalid solver-selection errors.

### 4.7 pipeline::Pipeline Named-Step Addressing

**Current state:** `Pipeline::set_param("<step_index>__<param>", value)` works; sklearn-style named-step addressing is deferred.

**Requirement:** Support `Pipeline::set_param("<step_name>__<param>", value)` and matching `get_params()` keys while preserving the existing index-prefixed routing.

**Implementation constraints:**

- Named steps are passed as `(std::string_view name, estimator)` tuples or through a helper equivalent to sklearn's `make_pipeline` naming convention.
- Duplicate names are rejected at construction.
- `GridSearchCV` and `RandomizedSearchCV` must accept both named and index-prefixed parameter grids.
- Existing positional pipeline construction remains valid.

**Tests:** Named parameter routing, duplicate-name rejection, nested pipeline routing, GridSearchCV integration, and backward compatibility for index-prefixed routing.

### 4.8 Exhaustive Non-Default Parity Fixtures

**Current state:** v1.1.0 parity covers every estimator at least once, mostly at default parameters.

**Requirement:** Expand the parity harness to cover meaningful non-default parameter combinations for all mature estimators.

**Coverage rules:**

- Every enum-valued parameter gets at least one non-default fixture.
- Every solver family gets at least one fixture.
- Every sparse-aware estimator gets one dense fixture and one sparse fixture where sklearn supports sparse input.
- Every random-state estimator gets a deterministic fixture with a fixed seed.
- Every estimator with class weights, sample weights, or multi-output support gets at least one fixture for that behavior.

**Implementation constraints:**

- Keep fixtures small and deterministic; CI must not regenerate Python data during normal test runs.
- Store fixture intent in `params.json` so failures explain which parameter branch regressed.
- The regenerator must continue to enforce `tests/parity/sklearn_version.txt`.

### 4.9 Benchmarks and Regression Sweep

**Current state:** Benchmark targets exist, but the v1.1.0 release sweep was deferred.

**Requirement:** Run and record a benchmark sweep covering v1.0.0 and v1.1.0 estimators before v1.2.0 release.

**Benchmark coverage:**

- Preprocessing: `StandardScaler`, `MinMaxScaler`, `Normalizer`.
- Linear models: `LinearRegression`, `Ridge`, `Lasso`, `ElasticNet`, `SGDClassifier`, `SGDRegressor`.
- Clustering: `KMeans`, `MiniBatchKMeans`.
- Ensembles: `RandomForest*`, `GradientBoosting*`, `HistGradientBoosting*`.
- Neural network: `MLPRegressor`, `MLPClassifier`.
- Sparse hot paths introduced in v1.1.0 and v1.2.0.

**Release rule:** Any >10% regression against the previous release runner result must be fixed, justified by a parity improvement, or explicitly accepted in `CHANGELOG.md`.

### 4.10 Manifold Algorithm Completeness

**Current state:** v1.1.0 shipped functional manifold learners, but some sklearn algorithm branches are simplified.

**Requirement:** Fill the highest-value missing algorithm branches.

| Estimator | v1.2.0 target |
|---|---|
| `TSNE` | Barnes-Hut acceleration for `method="barnes_hut"`; exact method remains available. |
| `LocallyLinearEmbedding` | `modified`, `hessian`, and `ltsa` method variants. |
| `MDS` | Multiple `n_init` restarts with best-stress selection. |
| `Isomap` | Optional truncated eigensolver path via §4.6. |
| `SpectralEmbedding` | Optional truncated eigensolver path via §4.6. |

**Tests:** Shape checks, trustworthiness / stress parity bands, deterministic `random_state`, and invalid parameter branches for each method.

### 4.11 Ensemble Parity Gaps

**Current state:** v1.1.0 shipped RandomForest, GradientBoosting, and HistGradientBoosting with documented simplifications.

**Requirement:** Close the most visible sklearn parity gaps in boosting and histogram boosting.

**GradientBoosting targets:**

- Regressor losses: `absolute_error`, `huber`, and `quantile`.
- Classifier losses: multiclass `log_loss` and binary `exponential`.
- Stochastic gradient boosting when `subsample < 1.0`.
- `oob_improvement_`, `train_score_`, and early-stopping attributes where sklearn exposes them.

**HistGradientBoosting targets:**

- Leaf-wise growth with `max_leaf_nodes`.
- Native histogram split-finding rather than sort-based fallback.
- Early stopping.
- Monotonic constraints.
- Native categorical features.
- Native sparse input path.

**Tests:** Loss-specific fixtures, multiclass classification, early-stopping behavior, monotonic constraint smoke tests, and sklearn parity bands for predictions and staged scores.

### 4.12 PCA Native Sparse Solvers

**Current state:** `PCA` accepts sparse input through a documented densify-and-fit fallback.

**Requirement:** Add native sparse-compatible `arpack` and `randomized` solver paths where sklearn permits sparse PCA input.

**Implementation constraints:**

- Use implicit centering so sparse input is not materialised dense.
- Reuse the randomized SVD machinery from `TruncatedSVD` where possible.
- Use §4.6 `Spectra` adapter when `svd_solver="arpack"` and the feature flag is enabled.
- Preserve dense `PCA` behavior unchanged.

**Tests:** Sparse/dense equivalence on centered fixtures, explained variance parity, transform / inverse-transform shapes, and solver-selection errors.

### 4.13 DecisionTreeRegressor Multi-Target Parity

**Current state:** `DecisionTreeRegressor::fit_multi` fits one independent tree per target column.

**Requirement:** Implement sklearn-style joint multi-target split scoring using the sum of target variances / MSE over all outputs.

**Implementation constraints:**

- Single tree structure shared by all targets.
- Leaf values are target vectors.
- Single-target behavior and accessors remain unchanged.
- Native sparse split-finding from §4.3 must support multi-target impurity calculations.

**Tests:** Correlated and anti-correlated target fixtures where independent trees diverge from sklearn, single-target backward compatibility, and parity for `RandomForestRegressor` multi-output behavior.

### 4.14 CalibratedClassifierCV Completeness

**Current state:** Binary calibration works; multiclass and `ensemble=false` are deferred.

**Requirement:** Add multiclass calibration and the non-ensemble refit path.

**Implementation constraints:**

- Multiclass calibration uses one calibrator per class, matching sklearn's one-vs-rest probability calibration.
- `ensemble=false` fits cross-validated calibrators, then refits the base estimator on all data.
- `classes_`, calibrated classifier storage, and `predict_proba` shapes match sklearn.

**Tests:** Binary and multiclass sigmoid calibration, isotonic calibration, `ensemble=true` vs `ensemble=false`, and wrappers around estimators exposing either `decision_function` or `predict_proba`.

### 4.15 Rendered Example Plot Gallery

**Current state:** The SkigenPlot rendering infrastructure exists and CI renders KMeans, t-SNE, and PCA clustering figures for guide pages.

**Registry audit:** `doc/api_registry.json` is already the right control point for this work. As of this planning pass it lists 65 public classes, 49 class examples, and only 3 plotted classes (`PCA`, `KMeans`, `TSNE`). The module-level `plot_generation` flag is enabled only for `Decomposition`, `Cluster`, and `Manifold`, while many other visually useful modules (`SVM`, `Ensemble`, `Calibration`, `FeatureSelection`, `Covariance`) are still unplotted.

**Requirement:** Expand rendered example plots across the documentation site.

The plot code should live in the same example source as the algorithm walkthrough. Each example remains a normal console example by default and gains an optional SkigenPlot branch guarded by `SKIGEN_EXAMPLE_WITH_PLOT`. The render target is only an orchestrator: it builds the plot-enabled examples, runs each registered target, and collects the generated dark/light images for the documentation site. This keeps the numerical example, sklearn-comparison snippet, and plotted visual story contained in one place per algorithm.

**Implementation constraints:**

- Keep plotting logic inside the relevant `examples/<module>/<algorithm>.cpp` source whenever practical, guarded with `#ifdef SKIGEN_EXAMPLE_WITH_PLOT`.
- Examples must still build and run without SkigenPlot when `SKIGEN_WITH_PLOT=OFF`.
- Register each plotted example with `skigen_enable_plot(<target> <stem>)` in `examples/CMakeLists.txt`.
- Register every rendered plot in the matching `doc/api_registry.json` class entry under `plots`, including `stem`, `title`, source `example`, and `snippet`.
- Store generated PNGs under `doc/website/static/img/examples/` at build time; do not commit generated PNGs.
- Embed figures through the generated docs' plot component so dark and light variants are both available.
- CI must run `skigen_render_example_plots` before `npm run build` for staging and main docs.
- CI must verify that every registry `plots[].stem` has both dark and light generated PNGs.

**Initial targets:** SVM decision regions, random forest feature importance, gradient boosting staged error, manifold embeddings, covariance ellipses, calibration reliability curves, and feature-selection score plots.

### 4.16 API Registry Coverage and Example Inventory

**Current state:** `doc/api_registry.json` is the single source of truth for generated API pages, sklearn reference links, guide links, examples, and rendered plot metadata. The file already blocks undocumented public classes at generation time, but its coverage metadata is behind the actual repository state: several guide pages and examples exist while the corresponding registry entries still have `documented=false`, no `guide`, or no `example` path.

**Requirement:** Promote the API registry from an API-page index to the authoritative release inventory for sklearn parity coverage.

**Required registry fields for every implemented public class:**

- `name` and `module`.
- `sklearn` and `sklearn_url` when the class has a sklearn or sklearn-adjacent reference.
- `documented=true` once a guide page exists.
- `guide` pointing to the Docusaurus guide route when one exists.
- `example` pointing to a buildable example source when one exists.
- `example_snippet` when a class-specific snippet is embedded in generated docs.
- `plots` for every rendered SkigenPlot figure, with `stem`, `title`, `example`, and `snippet`.

**Registry-backed release gates:**

- No implemented public class may ship without a registry entry.
- Every v1.2.0 class added or completed must update the registry in the same change as the header, tests, example, and guide.
- Existing registry entries must be reconciled with existing guide and example files before the v1.2.0 release candidate.
- The release checklist must include a registry audit that reports totals for implemented classes, documented classes, example-backed classes, and plotted classes.
- The documentation build must fail when a registry entry points at a missing guide, example source, plot snippet, or rendered plot asset.

**v1.2.0 registry backlog:**

- Mark existing guide-backed classes as documented and add missing `guide` routes for Bayesian/ARD regression, Naive Bayes, feature selection, isotonic, calibration, ensembles, SVM, neural-network, model-selection, and manifold entries.
- Add missing `example` paths for SVM, model-selection, and manifold classes that already have example sources under `examples/`.
- Expand `plots` beyond `PCA`, `KMeans`, and `TSNE`, prioritizing visually instructive sklearn-style examples.
- Add registry entries immediately for any accepted v1.2.0 modules such as `Impute`, `DiscriminantAnalysis`, `KernelRidge`, `Optimization`, or `QuantileRegressor` once their public headers exist.

## 5. Candidate New Component Specifications

> New families are candidates, not guaranteed release blockers. They should be accepted only after the §4 deferred work is either completed or deliberately moved to a later minor release.

### 5.1 cluster::{DBSCAN, OPTICS, HDBSCAN, Birch, MeanShift, AffinityPropagation, AgglomerativeClustering}

**sklearn reference:** `sklearn.cluster` unsupervised cluster estimators not covered by v1.0.0 / v1.1.0.

**Requirement:** Add the highest-value missing clustering estimators, prioritizing `DBSCAN`, `AgglomerativeClustering`, and `Birch`.

**Implementation constraints:**

- Reuse existing nearest-neighbor infrastructure for radius and k-neighbor queries.
- Sparse input support follows sklearn estimator-by-estimator behavior.

### 5.5 optimization::NelderMead

**sklearn reference:** No direct sklearn estimator/helper reference. This is a Skigen-native optimization utility. It is motivated by the Nelder-Mead simplex minimizer currently used in mne-cpp's sphere fitting code, not by scikit-learn's `QuantileRegressor` linear-programming solver names.

**mne-cpp cross-check:** `mne-cpp/src/libraries/math/simplex_algorithm.h` implements a header-only Nelder-Mead minimizer with reflection, expansion, contraction, and shrink steps. `mne-cpp/src/libraries/math/sphere.cpp` uses it to optimize the geometric sphere residual from an initial simplex around the point-cloud centroid. A Skigen replacement must match this nonlinear derivative-free use case, not scikit-learn's LP simplex wording.

**Requirement:** Add a high-performance, header-only Nelder-Mead optimizer suitable for reuse by Skigen estimators and by mne-cpp as a replacement for its local simplex implementation.

**Public API target:**

- Module umbrella: `#include <Skigen/Optimization>`.
- Primary type: `Skigen::optimization::NelderMead<Scalar>` or equivalent naming consistent with the final Skigen namespace style.
- Inputs: either an explicit `(n_parameters + 1) x n_parameters` initial simplex or a center vector plus per-dimension step sizes.
- Cost function: templated callable accepting an Eigen vector reference; no `std::function` on the hot path.
- Result type: best parameters, best cost, iteration count, evaluation count, convergence status, and final simplex diagnostics.

**Algorithm requirements:**

- Implement standard Nelder-Mead moves with configurable reflection, expansion, contraction, and shrink coefficients; defaults follow the classical coefficients from Nelder and Mead (1965).
- Support both relative function tolerance and absolute parameter-space tolerance.
- Enforce `max_iter` and `max_eval` separately.
- Preserve deterministic behavior for deterministic cost functions and initial simplexes.
- Accept `float` and `double`; do not assume a fixed dimension.

**Performance requirements:**

- Allocate all iteration workspace during setup; no heap allocation inside the main iteration loop.
- Use Eigen expressions for centroid and vertex operations while avoiding lazy-expression lifetime traps.
- Avoid full simplex sorting when only best, worst, and second-worst vertices are needed.
- Keep objective evaluations minimal: one reflection evaluation per iteration, then expansion / contraction / shrink only when required.
- Provide a benchmark against the current mne-cpp simplex on sphere fitting and at least one synthetic nonlinear objective.

**Replacement requirements for mne-cpp:**

- Provide an adapter example for the current sphere-fitting use case: centroid initialization, `simplex_size`, `ftol`, optional spatial tolerance, `max_eval`, and progress callback behavior.
- Validate that the Skigen optimizer recovers the same center/radius tolerance bands as the current mne-cpp tests on unit, offset, and noisy spheres.
- The implementation must be original Skigen code derived from the published Nelder and Mead algorithm, not copied from mne-cpp or Numerical Recipes.

**Tests:** Rosenbrock, quadratic bowl, flat / degenerate simplex behavior, max-evaluation termination, spatial-tolerance termination, float/double consistency, deterministic callback counts, and mne-cpp sphere-fitting parity fixtures.

### 5.6 linear_model::QuantileRegressor and Linear Programming Scope

**sklearn reference:** `sklearn.linear_model.QuantileRegressor`.

**Cross-check:** scikit-learn does not implement a reusable Nelder-Mead optimizer here. `QuantileRegressor` formulates quantile regression as a linear program and calls `scipy.optimize.linprog`; its `solver` parameter includes `highs`, `highs-ds`, `highs-ipm`, `interior-point`, and `revised simplex`. This is a separate LP problem from §5.5.

**Requirement:** Decide whether v1.2.0 should add `QuantileRegressor`. If accepted, add it as a linear-model estimator with an explicit LP backend strategy rather than treating it as the Nelder-Mead simplex feature.

**Implementation options:**

- Native Skigen LP backend: implement the quantile-regression LP directly, with a revised simplex or dual-simplex solver optimized for the structured equality matrix. This keeps Skigen dependency-free but is a substantial solver project.
- Optional external backend: support HiGHS behind `SKIGEN_ENABLE_HIGHS`, while keeping Eigen as the only required dependency and documenting that `QuantileRegressor` is unavailable or feature-gated without the backend.
- Deferred path: postpone `QuantileRegressor` until a broader LP backend is justified by more than one estimator.

**Estimator parity requirements if accepted:**

- Constructor parameters: `quantile`, `alpha`, `fit_intercept`, `solver`, and `solver_options` where representable in C++.
- Fitted attributes: `coef_`, `intercept_`, `n_iter_`, and `n_features_in_` equivalents.
- Sparse input: match sklearn's sparse support only for solver paths that actually preserve sparse CSC/CSR structure.
- Sample weights: implement zero-weight filtering and objective scaling to match sklearn.

**Tests:** Dense and sparse quantile fixtures, median regression with `quantile=0.5`, non-default quantiles, `alpha=0`, `fit_intercept=false`, sample weights including zeros, invalid quantile bounds, infeasible / iteration-limit behavior, and sklearn parity fixtures for the accepted solver backend.
