# Skigen v1.3.0 — Requirements Specification

**Status:** Draft
**Started:** 2026-06-22
**Predecessor:** [v1.2.0-requirements.md](v1.2.0-requirements.md)
**Standard:** EES (Energy-Efficient Software)

---

## 1. Release Theme

**EES focus:** Make Skigen *pipeline-complete* and broaden the estimator zoo into the high-value sklearn families that v1.2.0 deliberately deferred.

v1.0.0 broadened the surface, v1.2.0 made that surface deeper and more sklearn-exact (native sparse trees/PCA/HistGB, libsvm WSS3 SMO, Barnes-Hut t-SNE, QuantileRegressor, truncated eigensolvers). After v1.2.0, Skigen covers the core supervised, dimensionality-reduction, and clustering surface with parity verification, and in places (UMAP, native-sparse trees, HistGB feature-completeness) meets or exceeds the equivalent sklearn knobs.

The biggest remaining practical gap is **not** another solver refinement — it is the *data-shaping* surface that real pipelines depend on: categorical encoders, column-wise composition, and text vectorisation. v1.3.0 therefore leads with the `preprocessing` / `compose` / `feature_extraction` story, then adds the most-requested estimator families (ensemble meta-estimators, robust / GLM linear models, composable Gaussian-process kernels, and the next clustering tier).

### 1.1 Entry State (end of v1.2.0)

As of the v1.2.0 release candidate:

- **87 public classes across 24 modules**, with 47 parity cases verified against 44 sklearn reference fixture sets and 32 CTest targets.
- `doc/api_registry.json` is enforced by `doc/audit_api_registry.py` / the `skigen_audit_registry` CMake target; all 87 classes are documented, example-backed, and 8 are plot-backed.
- Native-sparse fit paths exist for `PCA`, `TruncatedSVD`, `DecisionTree*` (and the ensembles built on them), and `HistGradientBoosting*`.
- Shared internal helpers landed: `internal::randomized_svd`, `internal::HistTree`, `internal::solve_standard_lp` (interior-point LP), `internal::smo_binary_wss3` (libsvm WSS3), `internal::QuadTree` (Barnes-Hut), `internal::{smallest,largest}_eigenpairs` (+ optional `SKIGEN_ENABLE_SPECTRA`), and `internal::SparseFeatureColumns`.
- Deferred, non-blocking carry-overs from v1.2.0: record the §4.9 benchmark sweep on an unloaded machine; regenerate Python sklearn reference fixtures for the remaining default-only parameter branches (v1.2.0 §4.8); a sparse / large-`n` LP backend for `QuantileRegressor`.

### 1.2 Versioning Policy

Skigen still has a small adoption surface, so v1.3.0 keeps a controlled cleanup window open, but the bar rises: by 1.3.0 the public estimator API should be considered close to stable. Breaking changes remain allowed when they make the library more beautiful, coherent, sklearn-faithful, high-performance, or easier to maintain — but each breaking change must cite the parity, correctness, or maintainability reason it serves, and must be recorded in `CHANGELOG.md`.

Allowed breaking-change categories carry over from v1.2.0 (naming alignment with sklearn, signature corrections for parity, header relocation within a module, removal of pre-1.0 deprecation aliases). New estimators should be additive.

## 2. Inherited Guardrails

All non-negotiable rules from v1.0.0 through v1.2.0 remain in force:

- Header-only C++23 library; no compiled sources in `Skigen/`.
- Eigen remains the sole required numerical dependency. Optional backends (e.g. Spectra) stay behind feature flags guarded by `__has_include`.
- `Eigen::Ref<const MatrixType>` for dense read-only inputs and `Eigen::SparseMatrix` overloads where sklearn admits sparse data; **no silent densification** — a documented, tested fallback or a native sparse path.
- CRTP estimator hierarchy; no virtual dispatch on hot estimator paths.
- Strict sklearn parity for names, defaults, fitted attributes, error conditions, and deterministic random-state semantics.
- Zero warnings under `-Wall -Wextra -Wpedantic` on GCC >= 13, Clang >= 17, and MSVC >= 19.38.
- Every new estimator/transformer ships with unit tests, parity coverage, one runnable example, and a guide page, and is registered in `doc/api_registry.json`.

### 2.1 v1.3.0 Quality Gates

| Gate | Requirement |
|---|---|
| **Pipeline-first** | The encoder + `ColumnTransformer` story (§4) must land and be composable with the existing `Pipeline` before estimator-zoo work (§5) may block the release. |
| **Heterogeneous columns** | Transformers that select or transform a subset of columns must round-trip through `Pipeline` / `ColumnTransformer` with both index- and (where applicable) name-based column selection. |
| **Meta-estimator honesty** | Ensemble meta-estimators (Bagging/Voting/Stacking/AdaBoost) must reuse the existing base estimators via the CRTP interfaces without copying their fit logic, and must document any cloning / random-state-threading semantics. |
| **Parity depth** | Every new estimator adds at least one non-default parity fixture per meaningful solver / loss / kernel / strategy family. |
| **Sparse honesty** | Same as v1.2.0 — sparse overloads stay sparse or document + test the fallback. Encoders that produce sparse output (e.g. one-hot) must offer a sparse return path. |
| **Benchmark sweep** | The deferred v1.2.0 sweep is run and recorded for the v1.3.0 baseline; any >10% regression requires a fix or a documented exception. |
| **Documentation parity** | Guide pages state the sklearn reference class, supported parameter subset, parity fixtures, and known gaps. |
| **Maintenance review** | Architecture / maintainability / performance review after each major batch and before the release candidate. |

## 4. Pipeline-Shaping Surface (Priority)

### 4.1 preprocessing::OneHotEncoder and OrdinalEncoder

**sklearn reference:** `sklearn.preprocessing.OneHotEncoder`, `sklearn.preprocessing.OrdinalEncoder`.

**Requirement:** Encode integer-coded (and, where the C++ type system permits, string-coded) categorical columns into one-hot or ordinal representations.

**Implementation constraints:**

- `OneHotEncoder` must offer a **sparse output** path (CSR/CSC) as well as dense, matching sklearn's `sparse_output` default of sparse.
- Honour `categories="auto"` (learn from data) and explicit category lists; `handle_unknown` in `{"error", "ignore", "infrequent_if_exist"}` to the extent representable.
- `drop` in `{none, "first", "if_binary"}`; expose `categories_`, `drop_idx_`, and the inverse transform.
- `OrdinalEncoder` exposes `categories_`, `handle_unknown="use_encoded_value"`, and `unknown_value`.
- Skigen is numeric-first; string categories require a documented encoding-key convention (e.g. caller passes integer codes, or a `LabelEncoder`-style pre-step). Decide and document the string story explicitly rather than half-supporting it.

**Tests:** Dense and sparse one-hot equivalence, unknown-category handling, `drop` variants, inverse transform round-trips, and `Pipeline` integration.

### 4.2 compose::ColumnTransformer and FeatureUnion

**sklearn reference:** `sklearn.compose.ColumnTransformer`, `sklearn.pipeline.FeatureUnion`.

**Requirement:** Apply different transformers to different column subsets and horizontally concatenate the results — the keystone for heterogeneous tabular pipelines.

**Implementation constraints:**

- Column selection by index ranges / index lists; name-based selection only if a column-name convention is adopted project-wide (otherwise document index-only).
- `remainder` in `{"drop", "passthrough"}` (and an explicit transformer).
- Sparse-aware horizontal concatenation: if any sub-transformer emits sparse and the configured `sparse_threshold` is met, the combined output stays sparse.
- Must compose with `Pipeline` and participate in `get_params` / `set_param` routing using the existing named-step machinery.
- `FeatureUnion` is the all-columns special case; share the concatenation core.

**Tests:** Mixed numeric/categorical column pipelines, `remainder` behaviour, sparse concatenation, parameter routing through `GridSearchCV`, and fitted-attribute shapes.

### 4.3 feature_extraction::text Vectorisers

**sklearn reference:** `sklearn.feature_extraction.text.{CountVectorizer, TfidfTransformer, TfidfVectorizer, HashingVectorizer}`.

**Requirement:** Turn token-count / document inputs into the sparse feature matrices the v1.1.0/v1.2.0 sparse infrastructure was built for.

**Implementation constraints:**

- Skigen does not own a string-processing stack; the realistic scope is **count-based vectorisation from pre-tokenised input** (e.g. `std::vector<std::vector<std::string>>` or token-id streams) producing a sparse CSR document-term matrix, plus the TF-IDF reweighting (`TfidfTransformer`) on any sparse count matrix.
- `TfidfTransformer` parity: `norm` in `{"l1", "l2", none}`, `use_idf`, `smooth_idf`, `sublinear_tf`; expose `idf_`.
- `HashingVectorizer` uses the signed-hash trick into a fixed feature dimension (stateless, no vocabulary).
- A full regex tokenizer / analyzer pipeline is **out of scope**; document the tokenisation boundary clearly.

**Tests:** Count-matrix correctness on a small corpus, TF-IDF parity against sklearn `TfidfTransformer` on a fixed sparse count fixture, hashing-collision determinism, and `Pipeline` integration into a linear classifier.

### 4.4 preprocessing::KBinsDiscretizer, QuantileTransformer, PowerTransformer

**sklearn reference:** the like-named `sklearn.preprocessing` classes.

**Requirement:** Add the distribution-shaping transformers commonly used ahead of linear models.

**Implementation constraints:**

- `KBinsDiscretizer`: `strategy` in `{"uniform", "quantile", "kmeans"}`, `encode` in `{"ordinal", "onehot", "onehot-dense"}` (reusing §4.1), expose `bin_edges_`.
- `QuantileTransformer`: empirical-CDF mapping to uniform or normal output; `n_quantiles`, `output_distribution`, `subsample` with deterministic `random_state`.
- `PowerTransformer`: Yeo-Johnson (default) and Box-Cox with per-feature `lambda` fit by likelihood maximisation; optional standardisation.

**Tests:** Strategy-specific binning fixtures, inverse transforms, normal-output Shapiro-style sanity bands, and parity fixtures for the lambda estimates.

### 4.5 impute::KNNImputer and IterativeImputer

**sklearn reference:** `sklearn.impute.{KNNImputer, IterativeImputer}`.

**Requirement:** Add the two multivariate imputers that complement the existing `SimpleImputer`.

**Implementation constraints:**

- `KNNImputer`: nan-aware (masked) Euclidean distance, `n_neighbors`, `weights` in `{"uniform", "distance"}`.
- `IterativeImputer`: round-robin regression of each feature on the others until convergence or `max_iter`; pluggable base estimator (default a Ridge-style regressor), deterministic ordering, expose `n_iter_`.
- Both must preserve dense behaviour and document any sparse limitations.

**Tests:** Recovery of a known low-rank / linear fixture, neighbour-weighting behaviour, convergence on a correlated fixture, and `Pipeline` integration.

## 5. Estimator Zoo Expansion

### 5.1 ensemble Meta-Estimators

**sklearn reference:** `BaggingClassifier/Regressor`, `VotingClassifier/Regressor`, `StackingClassifier/Regressor`, `AdaBoostClassifier/Regressor`, `ExtraTreesClassifier/Regressor`.

**Requirement:** Add the generic ensemble meta-estimators and the Extra-Trees family.

**Implementation constraints:**

- Meta-estimators wrap one or more base estimators through the existing CRTP `Predictor` / `Classifier` interfaces; they must not duplicate base fit logic. Decide a cloning convention (the base config is copied per member; fitted state is per member).
- `Voting*`: `voting` in `{"hard", "soft"}`, optional weights; `Stacking*`: cross-validated out-of-fold meta-features feeding a final estimator.
- `AdaBoost*`: `SAMME`/`SAMME.R` for classification and `AdaBoost.R2` for regression, with sample-weight threading (requires base estimators to accept sample weights — gate / document where they don't).
- `ExtraTree*` adds random-threshold split selection to the tree core (reuse the existing split machinery with a random-cut policy); `ExtraTrees*` are the forest wrappers.

**Tests:** Accuracy/score bands vs sklearn on shared fixtures, soft-vs-hard voting, stacking out-of-fold correctness, AdaBoost weight evolution smoke tests, and Extra-Trees variance-reduction behaviour.

### 5.2 linear_model Robust and GLM Regressors

**sklearn reference:** `HuberRegressor`, `RANSACRegressor`, `TheilSenRegressor`, `PoissonRegressor`, `GammaRegressor`, `TweedieRegressor`.

**Requirement:** Add robust regression and the generalised-linear-model family.

**Implementation constraints:**

- `HuberRegressor`: IRLS / smooth-loss minimisation with `epsilon`, `alpha`; expose `scale_`, `outliers_`.
- `RANSACRegressor`: meta-estimator over a base regressor with `min_samples`, `residual_threshold`, deterministic `random_state`, `inlier_mask_`.
- `TheilSenRegressor`: median-of-slopes spatial-median estimator (subsampled for tractability).
- GLMs (`Poisson`/`Gamma`/`Tweedie`): exponential-family loss with log link, L2 penalty, IRLS or Newton solver; `TweedieRegressor` with `power` generalising the others.

**Tests:** Outlier-robustness fixtures (Huber/RANSAC/TheilSen recover the clean line under contamination), GLM parity on count/positive-continuous targets, and `power`-sweep behaviour for Tweedie.

### 5.3 gaussian_process Composable Kernels

**sklearn reference:** `sklearn.gaussian_process.kernels.*` and hyperparameter optimisation in `GaussianProcessRegressor/Classifier`.

**Requirement:** Turn the v1.2.0 fixed-hyperparameter GP into a composable, optimisable one.

**Implementation constraints:**

- Kernel objects: `RBF`, `Matern`, `ConstantKernel`, `WhiteKernel`, `DotProduct`, with `Sum` and `Product` composition and `+` / `*` ergonomics where C++ allows.
- Hyperparameter optimisation by marginal-likelihood maximisation (L-BFGS / gradient-based, feature-gated like the MLP LBFGS path if it reuses the same backend) with `n_restarts_optimizer`.
- Preserve the existing fixed-kernel API as a special case (a fully-specified kernel with optimisation disabled).

**Tests:** Kernel-composition correctness, marginal-likelihood gradient checks, recovery of known length-scales on a synthetic GP draw, and parity bands on a small fixture.

### 5.4 cluster Next Tier

**sklearn reference:** `SpectralClustering`, `OPTICS`, optionally `HDBSCAN`, `AffinityPropagation`.

**Requirement:** Add the highest-value clustering algorithms still missing after v1.2.0.

**Implementation constraints:**

- `SpectralClustering`: affinity graph → normalised Laplacian → truncated eigenvectors (reuse `internal::smallest_eigenpairs`) → KMeans on the embedding.
- `OPTICS`: reachability-ordering with `min_samples`, `max_eps`, and `xi` / `dbscan` cluster extraction.
- `HDBSCAN` is a stretch goal (mutual-reachability + condensed-tree); may defer.

**Tests:** Cluster-recovery on separable fixtures, deterministic `random_state`, eigensolver-path equivalence for SpectralClustering, and reachability-plot sanity for OPTICS.

### 5.5 naive_bayes Completion

**sklearn reference:** `ComplementNB`, `CategoricalNB`.

**Requirement:** Complete the naive-Bayes family.

**Implementation constraints:**

- `ComplementNB`: complement-class weighting for imbalanced text-style count data, `alpha`, `norm`.
- `CategoricalNB`: per-feature categorical likelihoods with `alpha` smoothing; expose `category_count_`, `class_count_`.

**Tests:** Imbalanced-count parity for ComplementNB, categorical-feature parity for CategoricalNB, and sparse-count support where sklearn provides it.

### 5.6 kernel_approximation and random_projection (Stretch)

**sklearn reference:** `Nystroem`, `RBFSampler`, `GaussianRandomProjection`, `SparseRandomProjection`.

**Requirement:** Add approximate feature maps that let linear models scale to kernel-like expressiveness.

**Implementation constraints:**

- `Nystroem` / `RBFSampler`: random Fourier / landmark features feeding a downstream linear model; deterministic `random_state`.
- Random projections: Johnson-Lindenstrauss dense and sparse projections with the standard density default.

**Tests:** Approximation-quality bands vs the exact kernel on a small fixture, deterministic seeds, and `Pipeline` integration.

## 6. Documentation Requirements

### 6.1 Requirements and Migration Notes

- This document is the source of truth for v1.3.0 scope; update it as items land or are explicitly rejected.
- `CHANGELOG.md` records every addition, parity fix, and remaining gap.

### 6.2 Guide Pages

Every new public class gets a guide page stating the sklearn reference, supported parameter subset, parity fixtures, and known gaps, plus one runnable example. Transformers that participate in `ColumnTransformer` / `Pipeline` must show a composed example.

### 6.3 API Reference

Regenerate API MDX pages for new modules (`Compose`, `FeatureExtraction`) and estimators. `doc/api_registry.json` must list every implemented public class with sklearn reference, guide route, example source, and rendered plot metadata where applicable, reconciled before MDX regeneration.

## 8. Out of Scope for v1.3.0

The following remain future work unless explicitly promoted into §4/§5 during planning:

- Full regex tokenisation / language-aware text analysis (only count-from-tokens + TF-IDF are in scope per §4.3).
- String-dtype / arbitrary object-dtype DataFrames; Skigen stays numeric-first with documented encoding conventions.
- HMM / time-series estimators.
- ONNX / PMML export and model serialisation formats.
- Dataset loaders (`fetch_*`, `load_*`).
- GPU offload via SYCL / CUDA / Metal.
- SIMD-explicit kernels beyond targeted benchmark-driven hot spots.
- `multiclass` / `multioutput` meta-estimators beyond what the native multi-output paths already cover.
- Any breaking API redesign not required to fix a documented sklearn parity bug.
